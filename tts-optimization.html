01

SGLang vs SGLang-Omni: TTS Is Not an LLM

SGLang serves one model with one compute pattern: a transformer that eats tokens and emits tokens. Higgs Audio v3 has that transformer inside it — but wrapped in a neural audio codec, a multi-codebook delay schedule, and a waveform decoder. Same backbone, completely different serving problem.

SGLANG · LLM INFERENCE

One model, one pattern

text in → text out
  • Single stage. Prefill + decode on the same transformer; one scheduler owns the GPU.
  • The token is the product. A sampled token goes straight to the user — no further compute needed.
  • Chunking is free. Text can be cut anywhere; streaming means flushing tokens as they sample.
  • One head, one vocabulary. Each step emits exactly one token.
  • Latency budget is dominated by the backbone forward; per-step overhead is real but secondary.
  • Memory model: one global KV-cache pool, one scheduler fraction.
SGLANG-OMNI · TTS INFERENCE

A pipeline, not a model

text + reference voice in → waveform out
  • Four heterogeneous stages. CPU preprocessing, codec encoder, AR engine, vocoder — different compute patterns, latency needs, memory behavior.
  • The token is an intermediate. Audio codes are worthless until a vocoder turns them into waveform — every request pays a second GPU model.
  • Chunking is dangerous. Naive splits produce audible clicks at every seam; the codec carries convolutional state across chunks.
  • Eight codebooks per step, offset by a delay pattern — a state machine, not a single argmax.
  • Latency budget is dominated by per-step overhead: 400–800 steps for 10s of speech amplify every microsecond.
  • Memory model: multiple models on one GPU must split memory by explicit contract.
The consequence: you cannot serve this by bolting an encoder and a vocoder onto an LLM server's request loop. The stages would serialize, queue behind each other, and fight for GPU memory. The fix is architectural — make the stages first-class and run them asynchronously.
02

Why Multi-Stage Async Wins

SGLang-Omni makes each stage an independent process with its own scheduler, connected by a Coordinator over split control/data planes. Stages stop waiting for each other — while request A is in the vocoder, request B is mid-decode and request C is being encoded.

BENEFIT / 01

Right scheduler per stage

The AR stage runs OmniScheduler with SGLang's continuous batching, KV-cache management and CUDA graph support; encoder and aggregator use a lean SimpleScheduler; streaming stages get StreamingSimpleScheduler. No stage drags another's machinery around.

BENEFIT / 02

Memory as a contract

Each stage on a GPU declares total_gpu_memory_fraction, and placement validation sums the budget per GPU before startup. Encoder, AR engine, and vocoder coexist on one H100 without silently starving each other.

BENEFIT / 03

Streaming-aware data movement

Control messages travel a ZMQ/msgpack control plane; tensors move on a relay data plane (shm, nccl, nixl, mooncake), with CUDA IPC for same-GPU chunks. Audio windows flow downstream while the AR loop is still generating.

One structure rules them all: the delay pattern

Codebook 0 carries coarse structure (pitch, intonation); codebook 7 carries high-frequency texture. Codebook i activates i steps after codebook 0 — a staircase.

It defines a four-phase per-request state machine — delay → active → wind-down (codebook 0 hits EOS) → finished — that the encoder must emit, the AR engine must advance every step, and the vocoder must invert. Each deep dive below collides with it.

codebook 0 active waiting on delay
04

Deep Dives, Stage by Stage

Issue/PR numbers link to the public roadmap, sglang-omni #478 — whose stated goal was "low latency and high throughput with no model performance reduction."

A

Encoder — the fastest compute is no compute

LRU cache · batched encoding · torch.compile

The encoder's profile: expensive (50–100ms), deterministic, and repetitive. The same reference voice produces the same tokens every time, and production users reuse a handful of voices across thousands of prompts. Profiling says: most encoder work is recomputation of known answers.

So the optimization order is: skip it (cache), amortize it (batch), compile it (static convolutional graph — the one place torch.compile genuinely fits).

arch  DAC-style codec, weights shared with vocoder
out  [T, 8] tokens → rearranged into delay pattern
key fact  deterministic per input → cacheable
LRULRU cache keyed by reference audio

Seen this voice before? Skip the encoder entirely and return the precomputed delay tokens. Every hit saves 50–100ms of GPU time and keeps the GPU free for AR decode — a double win under load.

#562#563#605
BATCHBatched encoder — with a production twist

Length-bucketed batch encoding lifts encoder throughput on paper. Measured in production, it turned GPU utilization from smooth into bursty spikes that fought the concurrent AR loops for SMs. Verdict: batched encoding was kept for offline warmup (pre-populating the voice cache at server start), while the online path encodes in isolation. Paper optimization ≠ production optimization.

#603#610
COMPILEtorch.compile on the encoder

Static shapes, a convolutional stack, no data-dependent branching — the encoder is the textbook torch.compile workload, and here it stays. (Spoiler: it did not survive on the AR backbone — see below.)

#612
B

AR Engine — CUDA graphs, radix cache, paged attention

Qwen3-4B backbone · fused 8-codebook head · 400–800 steps per 10s of speech

The heart of the system, and the main battleground. Each decode step: backbone forward → fused projection to eight codebook heads ([B, 8, V]) → sample eight tokens → advance the delay-pattern state machine → report to the CPU.

At 400–800 steps per request, the enemy is not FLOPs but per-step fixed cost: dozens of tiny kernel launches and device-to-host syncs. At ~0.1ms of overhead per step, that's ~80ms of pure waste per request — every microsecond removed here is multiplied by ~600.

Because the AR stage runs SGLang's OmniScheduler, it inherits two LLM-world weapons for free — paged attention and the radix cache — and adds the TTS-specific ones on top.

per-step  forward · 8-head projection · sample · D2H
state machine  delay → active → wind-down → finished
enemy  launch + sync overhead × ~600 steps
CONCEPTWhat a CUDA graph actually is

Normally the CPU launches every GPU kernel one by one, and each launch costs 5–10 microseconds of driver overhead. For big kernels that's noise; for an AR decode step made of dozens of tiny kernels, the CPU becomes the bottleneck — the GPU finishes work faster than the CPU can feed it.

A CUDA graph records the entire kernel sequence once, then replays the whole step as a single launch. The catch: the recording is frozen. Same kernels, same shapes, same memory addresses, every replay. Any data-dependent Python branch — if request.finished: — breaks the contract.

APPLYRewriting the delay pattern to fit the graph

The delay-pattern state machine was exactly what CUDA graphs forbid: per-request Python branching on delay counters, EOS detection, and wind-down logic. The rewrite turned control flow into data flow: every branch became a branchless, in-place tensor op on pre-allocated [max_batch, …] buffers — delay counters, EOC countdowns, finished flags, last-emitted codes — masked arithmetic instead of if/else, overwritten at fixed addresses every step.

Around the frozen graph, a runner copies active requests' state from the request pool into fixed "shadow" buffers, replays the graph, and scatters results back — all GPU-to-GPU, no host round-trip.

#503
GAINWhat it bought — and the two follow-ups

Graph replay collapses dozens of per-step launches into one, removing the per-step CPU overhead that was costing tens of milliseconds per request (~0.1ms × 400–800 steps). Two follow-ups finished the job: the three per-step D2H syncs were merged into one staging-tensor transfer (staging[:n].cpu()), and async one-step-lookahead decode exploited the fact that the token causal chain closes on the GPU — step t+1 launches before the CPU has processed step t, hiding host transfer entirely behind compute. Together these drive the single-request RTF to 0.147.

#564#572#590
RADIXRadix cache, partitioned by voice

SGLang's RadixAttention stores the KV cache of previous requests in a prefix tree, so a new request sharing a prefix skips that prefill. In TTS the shared prefix is the reference voice's tokens — so the cache is namespaced with an extra_key per reference audio. Requests using the same voice reuse its KV; different voices can never collide. The encoder's LRU and the radix cache stack: a repeated voice skips both the codec encode and the prompt prefill.

PAGEDPaged attention for free

KV memory is managed in fixed-size pages rather than contiguous per-request slabs — no fragmentation, no over-reservation, which is what lets 16 concurrent AR loops share one H100 alongside the encoder and vocoder. Inherited from the SGLang scheduler rather than rebuilt — the payoff of reusing OmniScheduler instead of a custom one.

#476
CUTtorch.compile on the backbone — removed

With manual CUDA graphs already eliminating launch overhead, compilation added only marginal throughput but a heavy cold-start penalty — so it was dropped. It also hid a trap: at CG batch ≥ 16 with compile enabled, WER exploded from 0.1–0.4% to 99–100% (an upstream SGLang × Inductor capture interaction; mitigated by capping _compiled_max_decode_bs=12). Correctness bugs in audio don't crash — they speak gibberish.

C

Vocoder — streaming without the clicks

DAC decoder · ~10ms per call · batched decode + windowed streaming

The vocoder inverts the codec: delay-pattern tokens back into waveform. Two distinct problems live here.

Problem one — queueing. Every AR loop converges on this stage. Under 16-way concurrency, finished requests pile up and decode serially: 16 × 15ms means the last request waits 240ms. Fixed by batching.

Problem two — streaming. Audio cannot be chunked like LLM text. The codec's transposed convolutions carry internal state across time; decode two chunks independently and the waveforms disagree at the boundary — an audible click at every seam. And mid-stream, the delay pattern's trailing rows are incomplete (high codebooks still filling); decode them and you inject noise.

stride  75 frames — accumulate, then decode
overlap  8 frames — re-decode boundary + crossfade
hold-back  4 frames — incomplete tail rows wait
floor  ≥ N rows to invert the pattern → TTFB ≈ 300–400ms
BATCHBatched vocoder decode

Collect requests inside a tight max_batch_wait_ms=2ms window, zero-pad to the batch max, stack as [B, num_codebooks, max_len], decode in one codec.decode_batch() call, then trim each waveform to its true length. The 240ms tail wait disappears.

User hears nothing until the entire utterance is generated and decoded. Simple and seam-free — but first audio arrives after seconds, unusable for conversation.

WINDOWED STREAMING

Decode in overlapping windows while the AR loop still runs. First audio at ~350ms; generation hides behind playback. The price: boundary management — solved below.

STREAMThe window protocol: stride / overlap / hold-back

Codes accumulate until the 75-frame stride triggers a decode. The decoder backtracks 8 frames into the previous window and re-decodes jointly, so the codec sees continuous context across the boundary; only the new samples past the overlap are emitted, blended with the previous chunk's tail by a linear crossfade envelope. The last 4 frames stay held back — their high codebooks are still filling — until the final flush. No clicks, no noise injection.

stride · 75 — decode trigger
overlap · 8 — crossfade
hold · 4 — tail

Irreducible startup cost: the vocoder needs at least N rows (N = number of codebooks) before it can invert the delay pattern and emit the first frame. Stacked with the stride, measured streaming TTFB ≈ 300–400ms — under the 500ms conversational threshold. You can verify this yourself in the demo below.
