01

An Assembly Line for Sound

Optimizing TTS looks like optimizing an LLM. It isn't. Higgs Audio v3 (~4B params, Qwen3 backbone) splits its end-to-end path into four stages with entirely different compute patterns, latency requirements, and memory behavior. Understanding how data moves through them is the prerequisite for everything that follows.

CPU · I/O BOUND

Preprocessing

Text tokenization and reference-audio loading. Unglamorous, but it shapes how requests queue before they ever touch a GPU.

tokenize + ref-audio load
GPU · STATIC COMPUTE

Audio Encoder

HiggsAudioCodec — a DAC-style neural codec — compresses the reference waveform into discrete tokens. 50–100ms per encode.

waveform → [T, 8] tokens
GPU · AUTOREGRESSIVE

TTS Engine

A Qwen3-based AR model generates multi-codebook tokens step by step. Eight codebook embeddings fused into a single lookup.

step-by-step → [B, 8, V]
GPU · FAST BUT QUEUE-PRONE

Vocoder

The DAC decoder turns tokens back into audible waveform. Shares weights with the encoder. ~10ms per call.

tokens → waveform @ 25fps

The Delay Pattern: The Hard Part

Codebook 0 carries coarse structure — pitch, intonation. Codebook 7 carries high-frequency texture. Codebook i activates i timesteps after codebook 0, forming a staircase.

This creates a four-phase state machine: delay → active → wind-down (codebook 0 hits EOS) → finished. Nearly every AR and vocoder optimization that follows had to be built around this state machine.

codebook 0 (structure) active waiting on delay
02

Profiling: Three Real Bottlenecks

Measure first, cut second. Profiling showed the bottleneck was never the "big model forward pass" — it was tiny overheads amplified by hundreds of steps.

BOTTLENECK / 01

AR Decode Dominates Everything

Ten seconds of speech takes 400–800 decode steps. Every step runs a backbone forward, head projection, sampling, and a device-to-host sync — 0.1ms of per-step overhead becomes ~80ms end to end.

400–800steps per 10s of speech · microseconds become milliseconds
BOTTLENECK / 02

The Encoder Is Expensive but Static

Each encode costs 50–100ms — yet in production, users reuse the same reference voice across many prompts. The same computation runs again and again for identical input.

50–100msper encode · repeated voices = repeated waste
BOTTLENECK / 03

Head-of-Line Blocking at the Vocoder

A single call takes only ~10ms, but under load multiple AR loops finish simultaneously: 16 requests at 15ms each, decoded serially, leave the last one waiting 240ms for no good reason.

240mstail wait @ 16 concurrent · fast kernels ≠ a fast system
03

Stage by Stage: The Surgery

Each stage got its own treatment: the encoder eliminates compute with caching, the AR engine eliminates overhead with CUDA graphs, the vocoder eliminates queueing with batching and windowing — and the whole thing crystallizes into a framework. Issue/PR numbers link to the public roadmap (sglang-omni #478).

Ⅰ

Preprocessing → fused into the encoder

CPU · tokenization + reference audio I/O

The CPU-side stage: tokenize input text, load and decode the reference audio. It contributes little compute but adds an extra inter-stage hop — and every hop means serialization, a queue, and another scheduling decision before the request ever reaches a GPU.

The fix here is structural, not algorithmic: collapse the stage entirely. Preprocessing was fused into the encoder stage, so each request crosses one boundary fewer.

role  text → token ids · audio file → waveform
cost  I/O-bound, negligible FLOPs
fix  stage fusion — fewer hops, less queueing
FUSEMerge preprocessing into the encoder stage

One fewer coordinator hop, one fewer queue, one fewer serialization boundary per request. The pipeline gets shorter before it gets faster.

#576
Ⅱ

Audio Encoder — the fastest compute is no compute

GPU · HiggsAudioCodec (DAC-style) · waveform → [T, 8] @ 25fps · 50–100ms per encode

The encoder converts a reference-voice waveform into discrete tokens across 8 parallel codebooks. The output is then rearranged into the delay pattern that the AR engine consumes as its voice-cloning prompt.

Its defining property: the computation is deterministic for a given input. Same reference audio in, same tokens out — every time. And in production, users overwhelmingly reuse a small set of voices across many prompts. That makes this stage a textbook caching target.

Where compute can't be skipped, it can still be made cheaper: the encoder is a static-shape convolutional network — exactly the workload torch.compile is good at, with no dynamic control flow to fight.

arch  DAC-style neural codec, weights shared with vocoder
out  [T, 8] tokens · rearranged into delay pattern
key fact  deterministic per input → cacheable
CACHELRU cache keyed by reference audio

If the same voice was seen before, skip the encoder entirely and fetch precomputed delay tokens — 50–100ms saved per cache hit, and the GPU stays free for AR decode.

#562#563#605
COMPILEtorch.compile on the encoder

Static shapes, convolutional stack, no branching — the encoder is the one part of the system where compilation actually pays off. (Contrast with the AR loop below, where it was removed.)

#612
DEMOTEDOnline batched encoding → offline warmup only

Length-bucketed online batch encoding raised throughput on paper. In production it turned GPU utilization from smooth into bursty spikes that fought the concurrent AR decode loops for resources. Batched encoding was demoted to offline server warmup; the online path stays isolated. Paper optimization ≠ production optimization.

#603#610
OPENEncoder CUDA graph

Capturing the encoder forward into a CUDA graph is still on the roadmap — the remaining launch overhead on cache misses.

Ⅲ

AR Engine — microsecond surgery

GPU · Qwen3-4B backbone · fused 8-codebook head [B, 8, V] · 400–800 steps / 10s speech

The heart of the system. Each decode step runs the backbone forward, projects to eight codebook heads at once (embeddings fused into a single lookup producing [B, 8, V]), samples eight tokens, advances the delay-pattern state machine, and reports back to the CPU.

At 400–800 steps per request, the enemy is not FLOPs — it is per-step fixed cost: kernel-launch latency from dozens of tiny kernels, and pipeline stalls from device-to-host synchronization. Every microsecond removed here is multiplied by ~600.

The campaign ran as four moves, each attacking one class of overhead — and ended with a deliberate removal and a debugging war story.

state machine  delay → active → wind-down → finished
per-step  forward · 8-head projection · sample · D2H
enemy  launch overhead × ~600 steps, not FLOPs
MOVE 1Capture the decode loop in a CUDA graph

A CUDA graph replays a fixed kernel sequence — any data-dependent Python branch breaks the recording, and the delay-pattern state machine was full of them. The fix: rewrite the state machine as in-place tensor ops on fixed addresses. Per-request decode state (delay counters, EOC countdowns, finished flags, last-emitted codes, sampled outputs) is pre-allocated once as [max_batch, …] buffers and overwritten in place each step. Around the captured graph, the runner copies active rows from the request pool into fixed "shadow" buffers, lets the graph read and update them in place, then scatters results back — all GPU-to-GPU.

#503
MOVE 2Merge three D2H syncs into one

The baseline did three separate device-to-host syncs per step to check tokens and state — three pipeline stalls. All intermediate results now land in a single staging tensor (_cg_collect_staging) and cross to the CPU in one combined transfer: staging[:n].cpu().

#564#572
MOVE 3Async one-step-lookahead decode

The key insight: the token causal chain closes entirely on the GPU. Step t+1 doesn't need the CPU's processing of step t's samples — so the GPU launches it immediately. A GPU-side sampler pool ([max_bs+1, 8]) holds each request's latest codes; the CPU reads last step's output from host staging, fills row tables and sampling params, and the two sides communicate through shared ping-pong buffers. The GPU never waits; host transfer cost is fully hidden.

#590
CUTtorch.compile on the backbone — evaluated, then removed

With manual CUDA graphs already eliminating launch overhead, compilation offered only marginal throughput — while adding a heavy cold-start compile penalty. It was removed as a pragmatic trade: fast initialization over redundant runtime optimization.

#565#579
War story — the 100% WER cliff: at CUDA-graph batch sizes ≥ 16, WER outliers went catastrophic — 99–100% WER versus a 0.1–0.4% baseline — when torch.compile was enabled on the backbone at bs == cuda_graph_max_bs. Diagnosis traced it to an upstream SGLang × Inductor CUDA-graph-capture interaction. Mitigation: cap _compiled_max_decode_bs = 12 until the upstream fix lands. Correctness bugs in inference systems don't always crash — sometimes they just speak gibberish. (#578, #579)
Ⅳ

Vocoder — batching + windowed streaming

GPU · DAC decoder, weights shared with encoder · ~10ms per call · 25fps output

The vocoder inverts the codec: delay-pattern tokens back into waveform. Fast per call — but it sits at the convergence point of every AR loop, so concurrency turns it into a serial chokepoint.

Streaming is where it gets genuinely hard. You cannot chunk audio codes the way you chunk LLM text: a neural codec carries internal convolutional state, and decoding chunks independently produces an audible click at every seam. Worse, the delay pattern means any mid-stream snapshot has incomplete trailing rows — the higher codebooks haven't caught up yet — and decoding them injects noise.

The solution is a carefully tuned window protocol: accumulate, re-decode the boundary, crossfade, and hold back what isn't ready.

stride  75 frames — accumulate before decoding, amortize launches
overlap  8 frames — re-decode into the previous window, crossfade
hold-back  4 frames — keep incomplete tail rows until final flush
floor  ≥ N rows needed to invert the pattern → TTFB ≈ 300–400ms
BATCHBatched vocoder decode

Requests are collected inside a tight max_batch_wait_ms=2ms window, zero-padded to the batch max length, stacked as [B, num_codebooks, max_len], and decoded in one codec.decode_batch() call. Each waveform is then trimmed back to its true length (length × samples_per_frame). The 240ms tail wait disappears.

#569#574
STREAMWindowed streaming with overlap + crossfade

Codes accumulate until the 75-frame stride triggers a decode. The decoder backtracks 8 frames into the previous window and re-decodes jointly, so the codec sees continuous context across the boundary. Only the new samples past the overlap are emitted, blended with the previous chunk's held tail using a linear fade-in/fade-out envelope — smoothing any residual amplitude mismatch. Four trailing frames stay held back (their high codebooks are still filling) until the final flush. No clicks, no noise injection.

#597#614
stride · 75 frames — decode trigger
overlap · 8 — re-decode + crossfade
hold · 4 — incomplete tail
Irreducible startup cost: the delay pattern means the vocoder needs at least N rows (N = number of codebooks) before it can invert the pattern and emit the first audio frame. Stacked with the stride, measured streaming TTFB lands at ~300–400ms — comfortably under the 500ms conversational threshold.
Ⅴ

The System Layer — crystallized into SGLang-Omni

Coordinator · per-stage schedulers · memory contracts · split control/data planes

The per-model surgery generalized into a framework for multi-stage heterogeneous generation. A config statically declares pipeline stages, GPU placement, and process topology; a Coordinator routes requests between stages; each Stage is an I/O shell that receives upstream data, drives its internal scheduler, and streams output downstream.

The design principle: new models should be integrated by declaring a topology and plugging in hooks — not by threading special-case branches through the framework. The same machinery now serves Fish Audio S2-Pro, Voxtral TTS, Qwen3-TTS, MOSS-TTS, and omni models like Qwen3-Omni, Ming-Omni, and LLaDA2.0-Uni.

refactor  custom HiggsScheduler dropped → shared OmniScheduler
principle  declare topology + hooks, not special branches
serves  5 TTS models + 3 omni models and counting
SCHEDOne scheduler interface, three implementations

The AR stage runs OmniScheduler — inheriting SGLang's continuous batching, mixed prefill/decode scheduling, KV-cache management, RadixAttention, and CUDA graph support. Encoder and aggregator stages use SimpleScheduler; streaming stages use StreamingSimpleScheduler. The custom HiggsScheduler was deleted in favor of the shared one.

#476
CACHERadix cache partitioned by reference voice

RadixAttention prefix caching is namespaced with an extra_key per reference audio — requests sharing a voice reuse prefix KV; requests with different voices never collide.

MEMGPU memory as a stage-level contract

Each stage on a GPU declares total_gpu_memory_fraction; placement validation sums the budget per GPU before startup. Memory is a negotiated contract between stages, not a single global scheduler fraction — so stages can't silently starve each other.

COMMSplit control and data planes

Control messages (submit, data-ready, stream, complete, abort) travel over a ZMQ/msgpack control plane. Tensor payloads move over a relay data plane with pluggable backends — shm, nccl, nixl, mooncake — with local dispatch for same-process edges and CUDA IPC for same-GPU streaming chunks. A true SSE streaming scheduler pushes first-audio latency down further.

06

Six Engineering Lessons

01

Profile before you cut

The bottleneck was never the big forward pass — it was tens of microseconds of launch and sync overhead amplified by hundreds of steps. Intuition lies; the profiler doesn't.

02

Attack each layer with its own weapon

The encoder eliminates compute (LRU cache), the AR engine eliminates overhead (CUDA graphs + merged syncs + async lookahead), the vocoder eliminates queueing (batching + windowing). One diagnosis per stage, one cure per stage.

03

Respect the domain

Audio is not text: delay-pattern state machines, seam clicks at chunk boundaries, noise from incomplete codebook rows. Optimizations must be built for the signal — naively porting LLM tricks causes acoustic accidents.

04

Dare to remove things

Online batched encoding was demoted; backbone torch.compile was deleted. Both calls were made on production measurements, not paper gains. Deleted code is an optimization too.

05

Hold the quality line

The roadmap's stated goal: low latency and high throughput with no model performance reduction. When CG batch ≥ 16 with compile drove WER from 0.4% to 100%, the team capped batch size rather than ship the speed. Correctness bugs in audio don't crash — they speak gibberish.

06

Crystallize into a framework

The endgame is generalization: SGLang-Omni turns per-model surgery into declare a topology + plug in hooks. The same machinery now serves Fish Audio S2-Pro, Qwen3-TTS, MOSS-TTS, Qwen3-Omni, and more.
