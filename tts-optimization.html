01

一条声音的流水线

优化 TTS 看似与优化 LLM 相同,实则瓶颈截然不同。Higgs Audio v3(约 4B 参数,Qwen3 骨干) 的端到端路径被切成 四个计算模式完全不同的阶段——理解数据如何穿过它们,是一切优化的前提。

CPU · I/O 密集

预处理

文本分词与参考音频加载。看似平凡,却决定请求进入 GPU 前的排队形态。

tokenize + ref-audio load
GPU · 静态计算

音频编码器

HiggsAudioCodec(DAC 风格神经编解码器)把参考波形压成离散 token,单次 50–100ms。

waveform → [T, 8] tokens
GPU · 自回归主干

TTS 引擎

基于 Qwen3 的 AR 模型逐步生成多码本 token,8 个码本的嵌入融合为单次查找。

step-by-step → [B, 8, V]
GPU · 快但易排队

声码器

DAC 解码器把 token 还原为可听波形,与编码器共享权重,单次调用约 10ms。

tokens → waveform @25fps

延迟模式:最棘手的部分

码本 0 负责粗结构(音高、语调),码本 7 负责高频纹理。码本 i 相对码本 0 延迟 i 个时间步激活——形成一个阶梯。

它构成一个四阶段状态机:延迟 → 活跃 → 收尾(码本 0 遇 EOS)→ 完成。后续几乎所有 AR 与声码器优化,都必须绕着这个状态机做。

码本 0(结构) 已激活 等待延迟
02

Profiling:三处真正的瓶颈

先测量,再动刀。性能分析揭示的瓶颈不在"大模型前向"本身,而在被数百步放大的微小开销。

BOTTLENECK / 01

AR 解码主导一切

10 秒语音需要 400–800 个解码步。每步都有前向、头投影、采样与 D2H 同步——每步 0.1ms 的开销,端到端就是约 80ms。

400–800steps / 10s speech · 微秒被放大为毫秒
BOTTLENECK / 02

编码器贵,但内容静态

单次编码 50–100ms;而生产环境中,用户往往用同一个参考音色发送多个请求——同样的计算被反复执行。

50–100msper encode · 重复音色 = 重复浪费
BOTTLENECK / 03

声码器队头阻塞

单次调用仅约 10ms,但高并发下多个 AR 循环同时完成:16 个请求各 15ms 串行解码,最后一个要白等 240ms。

240mstail wait @16 并发 · 快内核 ≠ 快系统
03

逐层手术:优化历程

编码器靠缓存消灭计算,AR 解码靠 CUDA Graph 消灭开销,声码器靠批处理与窗口化消灭排队——最后沉淀为框架。

P1
Phase 1 · Encoder

编码器:最快的计算是不计算

引入 LRU 缓存存储参考音频的编码结果:同一音色再次出现时直接跳过编码器,立即取回预计算的延迟 token。

一次反向决策:按长度分桶的在线批量编码,纸面上提升吞吐,实测却让 GPU 利用率从平稳变成间歇尖峰,与并发 AR 解码激烈争抢资源。最终把批量编码退役为离线预热专用,在线路径保持隔离——纸面优化 ≠ 生产优化。
P2
Phase 2 · AR Decode

AR 解码:微秒级的四步推进

核心战场。每步几十微秒的 kernel launch 与同步开销,在数百步里被无情放大——必须逐一拔除。

MOVE 1CUDA Graph 化整个解码循环

难点:Graph 要求执行路径完全静态,而延迟模式状态机满是 Python 分支。解法:把状态机改写为固定地址上的原位张量操作——预分配 [max_batch, …] 的延迟计数器、EOC 倒计时、完成标志等,每步原地覆盖;runner 把活跃请求状态 GPU→GPU 拷入"影子"缓冲区,Graph 重放后再散布回请求池。

MOVE 2合并 D2H 同步

基线每步做 3 次独立的 device-to-host 同步检查 token 与状态,3 次流水线停顿。改为把所有中间结果打包进单个 staging 张量,每步只做一次合并传输:staging[:n].cpu()。

MOVE 3异步一步先行(lookahead)解码

关键洞察:token 因果链可完全在 GPU 上闭合——无需等 CPU 处理第 t 步的采样结果,GPU 直接启动第 t+1 步。GPU 侧采样器池 + CPU/GPU ping-pong 共享缓冲区,单步流水重叠,主机传输开销被完全掩盖。

CUT评估后移除 torch.compile

手动 CUDA Graph 已消除绝大部分 launch 开销,torch.compile 只剩边际收益,却带来巨大的冷启动编译惩罚。果断移除——优化也包括做减法。

P3
Phase 3 · Vocoder

声码器:批处理 + 窗口化流式

批量解码:在 max_batch_wait_ms=2ms 的小窗口内攒请求,零填充到批内最大长度,一次 decode_batch() 解掉,再按 length × samples_per_frame 裁回真实长度。尾延迟问题解决。

窗口化流式则难得多:神经编解码器在每个拼接边界会产生可听的"咔哒"声(卷积内部状态被切断);延迟模式下中途快照的尾行码本不完整,直接解码会注入噪声。不能照搬 LLM 的分块直觉。

stride · 75 帧(≈1s)攒够才解码
overlap · 8 帧回溯重解 + 交叉淡化
hold-back · 4 帧扣住未填满的尾行
不可压缩的启动成本:延迟模式决定了声码器至少需要 N 行(N = 码本数)才能反转 pattern、产出第一帧音频。叠加 stride 后,实测流式 TTFB ≈ 300–400ms,仍低于 500ms 的会话体验阈值。
P4
Phase 4 · System

框架化:沉淀为 SGLang-Omni

把单模型的手术经验泛化为多阶段异构生成框架:配置静态声明流水线阶段、GPU 放置与进程拓扑,Coordinator 负责阶段间路由。

SCHED每阶段独立选调度器

AR 阶段用 OmniScheduler(继承 SGLang 连续批处理、KV 缓存、RadixAttention、CUDA Graph);编码器/聚合器用 SimpleScheduler;流式阶段用 StreamingSimpleScheduler。

CACHE按参考音频分区的前缀缓存

RadixAttention 以 extra_key 命名空间按参考音频分区,同音色请求复用前缀 KV。

MEM显存即阶段级资源合约

每个阶段声明 total_gpu_memory_fraction,启动前按 GPU 求和校验预算——而不是交给全局调度器的一个分数,避免阶段间打架。

COMM分层通信平面

控制消息走 ZMQ/msgpack 控制面;张量走 relay 数据面(shm / nccl / nixl / mooncake);同 GPU 走 CUDA IPC。SSE 真流式调度器进一步压低首音频延迟。

05

五条工程启示

壹

先测量,再动刀PROFILE FIRST

瓶颈不在"大模型前向",而在每步几十微秒的 launch 与同步开销 × 数百步的放大效应。直觉会骗人,profiler 不会。

贰

逐层各个击破LAYER BY LAYER

编码器用缓存消灭计算,AR 解码用 CUDA Graph + 合并同步 + 异步先行消灭开销,声码器用批处理 + 窗口流式消灭排队。

叁

尊重领域特性DOMAIN MATTERS

音频不是文本:延迟模式状态机、拼接点击声、码本不完整噪声——优化必须为音频量身定制,照搬 LLM 套路会出声学事故。

肆

敢做减法DARE TO CUT

放弃在线批量编码、移除 torch.compile,都基于生产实测而非纸面收益。删掉的代码也是优化。

伍

沉淀为框架GENERALIZE

从单模型手术升级为 SGLang-Omni 通用多阶段框架:新模型接入 = 声明拓扑 + 插钩子。已支持 Fish Audio S2-Pro、Qwen3-TTS、MOSS-TTS、Qwen3-Omni 等。
