Slide 11 · LLM vs TTS~55s · set the thesis

TTS Inference Is Not LLM Inference

Thanks — I'll take the inference part. And I want to start with the punchline up front: TTS inference is not LLM inference, even when there's an LLM inside.

LLM inference is one dominant decode loop. One model, one scheduler, text in, text out. The token you sample is the product — you flush it to the user and you're done. The whole system optimizes time-to-first-token, throughput, and KV reuse.

TTS inference is a multi-stage audio-generation pipeline: preprocess, then a reference-audio encoder, then the AR engine, then a vocoder. The token you sample is just an intermediate — it's worthless until a second GPU model turns it into a waveform. The stages have different compute patterns, different latency budgets, and may want different schedulers. And the goal shifts: you're balancing real-time latency, smooth streaming, and audio quality — not just maximizing text serving efficiency.

Slide 12 · SGLang-Omni Overview~60s

Declare Stages, Choose Schedulers

So we built the serving layer on SGLang-Omni, and the design philosophy fits on one line: declare stages, choose schedulers, define topology — and let the runtime handle routing, streaming, data movement, and resource isolation.

A Coordinator routes requests across the four stages. Each stage picks the scheduler that fits its job: preprocessing gets a lightweight CPU-side scheduler; the encoder gets a lean single-threaded one; the AR engine runs the full OmniScheduler — which inherits SGLang's continuous batching, KV cache and tree cache, and CUDA graph support; and the vocoder gets a streaming-aware Code2Wav scheduler with two execution modes — batched full-WAV, or streaming chunks over SSE.

Underneath: a ZMQ control plane for messages, a relay data plane for tensors — shared memory, NCCL, or CUDA IPC — and per-stage GPU memory budgets declared up front, so colocated stages can't starve each other.

One detail I love: PR 476 deleted our bespoke HiggsScheduler and replaced it with the shared one. Less custom code, more inherited optimization.

Slide 13 · Disaggregated Stages~40s

Same Request = Ordered Stages

Why stages at all? This slide is the answer.

If one coupled worker owns the whole pipeline end to end, request two waits for all of request one — even when the encoder is free and only the AR engine is busy. Disaggregate the stages into independent queues, and each request still flows in order — preprocess, encode, generate, vocode — but different requests overlap freely: while request one is in the vocoder, request two is mid-decode and request three is being encoded.

Same per-request order. Less blocking, better scheduling, higher utilization — one slow stage stops blocking everything.

Slide 14 · Preprocessing~35s

Stage 1: Preprocessing

Now one slide per stage, in pipeline order.

Preprocessing parses the speech request, normalizes text and inline control tokens, validates the reference audio, and builds the stage-native request objects. Almost zero FLOPs — it's pure CPU and I/O work. The profiling insight: its real cost is boundary overhead, not compute.

So it runs on a CPU-side thread pool that preps several requests in parallel, and the encoder path can absorb nearby preprocessing work to cut a stage boundary. Simple requests with no reference audio take a fast path. Net effect: the GPU never waits on I/O — by the time it finishes one request, the next is already prepped.

Slide 15 · The Encoder~50s

Stage 2: The Reference-Audio Encoder

The reference-audio encoder converts your voice sample into speaker-conditioning tokens. It costs 50 to 100 milliseconds — and here's the profile finding: it's deterministic and it's repetitive. Users reuse the same handful of voices across thousands of prompts. Most encoder work is recomputing answers we already know.

So, three moves. An LRU cache keyed by the reference audio — repeat voices skip the encoder entirely. Batched encoding for utilization — though we learned a production lesson there: online batching caused GPU contention with the latency-critical decode loops, so we keep the batch win for offline warmup and encode in isolation online. And torch.compile, which is a genuinely good fit here — static shapes, a convolutional stack, no branching.

Key idea: the encoder stage becomes much cheaper when reuse, batching, and boundary reduction are handled explicitly.

Slide 16 · The AR Engine~100s — THE CORE SLIDE

Stage 3: CUDA Graphs & Async Decode

The AR engine is the dominant stage, so this is where the heavy machinery goes. Ten seconds of speech is 400 to 800 decode steps, and each step's overhead multiplies by that count. A tenth of a millisecond per step is 80 milliseconds per request, gone.

Two foundations come free from reusing the shared scheduler: paged attention, which lets sixteen concurrent decode loops share one H100 without fragmenting KV memory, and the radix cache, namespaced per reference voice, so repeated voices skip the prompt prefill — it stacks with the encoder's LRU.

The biggest lever is CUDA Graphs. Without them, the CPU launches every small kernel one by one — each launch costs 5 to 10 microseconds, and the GPU starves between launches. A CUDA graph captures the whole decode step once and replays it with a single cheap launch. The catch: the capture is frozen — no data-dependent branching — so we rewrote the delay-pattern state machine as branchless, in-place tensor operations on fixed buffers. Control flow became data flow.

And then one more bubble: every step ends with a heavy host-side collect — sampler-state scatter, delay-pattern bookkeeping, end-of-stream handling — in Python, every step, with the GPU idle through it. The fix is async decode: a launch-first, one-step lookahead. We split the step into a launch half — forward, on-GPU sampling, an async copy into a pinned ping-pong buffer — and a resolve half, the host collect. The loop launches step N, then resolves step N minus one, so the previous step's collect hides under the current forward.

Output is bit-identical with it on or off — and on the full benchmark at concurrency sixteen it's worth another 12.7% throughput, 16% off mean latency, and the tail RTF, p99, drops by almost 40 percent. The longer the request, the more it pays.

Slide 17 · The Vocoder~60s

Stage 4: Streaming Without the Clicks

Last stage: the vocoder — codebook tokens to 24-kilohertz waveform. Fast per call, about ten milliseconds, but it has two traps.

Trap one: queueing. Every decode loop converges here, so under load it becomes a tail-latency source. Fix: batched vocoder decode — collect requests in a tight window, decode as one batch.

Trap two: streaming. You cannot chunk audio the way you chunk text — the codec carries convolutional state across time, and naive chunk boundaries produce an audible click at every seam. So the streaming path decodes in overlapping windows: accumulate codes to a stride, backtrack into the previous window and re-decode jointly, crossfade the overlap, and hold back the final frames whose codebooks haven't filled yet. The chunk and done lifecycle is managed by the streaming scheduler, with real SSE output.

Vocoder performance is not only raw decode speed — it's batching, streaming lifecycle, and output smoothness. First audio lands in roughly 350 milliseconds, well under the conversational threshold.

Slide 18 · Performance~45s · land the numbers

What Does All of That Buy?

One H100, full Seed-TTS English set.

At concurrency one: 617 milliseconds mean latency, real-time factor 0.147 — that's seven times faster than real time. Scale to concurrency sixteen: 14.7 requests per second, generating almost 62 seconds of audio every second, and the real-time factor is still only 0.26. Sixteen times the load, latency grows less than two-x. That's the multi-stage async design absorbing load.

And the guardrail throughout the whole roadmap was: no model performance regression. Speed was never traded for WER. That's the inference story — happy to take questions.

Inference Q&A

Is async decode safe — doesn't lookahead change outputs?

No — only the host collect is deferred one step; the token causal chain stays on the GPU. Outputs verified bit-identical ON vs OFF (greedy, bs=1 and bs=4, 100/100). bs=1 takes a sync fast path since there's nothing to overlap — break-even by design.

Why not torch.compile on the AR backbone too?

Manual CUDA graphs already removed the launch overhead, so compile added only marginal throughput but a heavy cold-start penalty. It also exposed an upstream Inductor × CUDA-graph-capture interaction at batch ≥ 16 that drove WER to ~100% — we capped the compiled batch size and dropped it from the backbone. It stays on the encoder, where it genuinely fits.

How does the delay pattern survive CUDA graph capture?

The state machine was rewritten as branchless, in-place tensor ops on pre-allocated fixed-address buffers — masked arithmetic instead of if/else. The graph replays the same kernel sequence; per-request state is swapped in and out through shadow buffers, GPU-to-GPU.

Why is batched encoding offline-only?

In isolation it's a real throughput win. Online, encode bursts grabbed SMs from the latency-critical decode loops — per-encode cost down, end-to-end tails worse. So the batch win is harvested at server warmup to pre-fill the voice cache; the online path encodes in isolation.

What bounds streaming TTFB?

An irreducible floor: the vocoder needs at least N rows (N = codebooks) to invert the delay pattern, plus the window stride. Measured TTFB is ~300–400ms — under the 500ms conversational bar.

Does this scale beyond one GPU?

Stages are declared with explicit GPU placement and per-stage memory budgets; TP expansion applies to the AR stage, and the data plane already speaks NCCL/shm/CUDA-IPC. Cross-node multi-stage pipelines are on the roadmap.
